1️⃣ 题意理解

核心要求： 统计截至 2019-07-27（包含当天）的近 30 天内，每天的活跃用户数。只要用户在某天有任意活动记录，就视为该天的活跃用户。

关键约束： 1. 时间范围：activity_date 必须在 2019-07-27 之前的 30 天内（包含 2019-07-27） 2. 活跃定义：当天有任意 activity_type 记录即算活跃 3. 去重：同一用户在同一天可能有多个活动，只计一次 4. 输出：只输出有活跃用户的日期，按任意顺序

边界情况： - 日期范围可能为空（无符合条件的数据） - 同一天同一用户有多个活动需要去重 - 日期可能不连续，只输出有活跃用户的日期

输入输出： - 输入：Activity 表，包含 userid, sessionid, activitydate, activitytype - 输出：包含两列的表：day（日期）, active_users（活跃用户数）

2️⃣ 解题思路

解法一：基础筛选 + 分组统计

核心思想： 1. 先筛选出符合日期范围的数据 2. 按日期和用户去重 3. 按日期分组统计用户数

步骤： 1. WHERE 子句筛选日期范围：activity_date BETWEEN '2019-06-28' AND '2019-07-27' 2. 使用 DISTINCT 或 GROUP BY 对 (activitydate, userid) 去重 3. 按 activitydate 分组，COUNT(DISTINCT userid) 或 COUNT(*) 统计 4. 重命名列并输出

注意事项： - 日期范围计算：2019-07-27 往前推 29 天是 2019-06-28（共30天） - 使用 DATESUB 或 DATEADD 函数确保日期计算准确

解法二：子查询去重 + 分组

核心思想： 先通过子查询获取去重后的每日活跃用户列表，再分组统计

步骤： 1. 子查询中筛选日期并去重 2. 外层查询按日期分组统计 3. 更清晰但可能效率略低

3️⃣ 代码实现

解法一：直接筛选分组

// SQL 实现（题目要求 SQL 查询）
SELECT 
    activity_date AS day,
    COUNT(DISTINCT user_id) AS active_users
FROM Activity
WHERE activity_date BETWEEN DATE_SUB('2019-07-27', INTERVAL 29 DAY) AND '2019-07-27'
GROUP BY activity_date
ORDER BY activity_date;

解法二：子查询方式

// SQL 实现
SELECT 
    day,
    COUNT(*) AS active_users
FROM (
    SELECT DISTINCT
        activity_date AS day,
        user_id
    FROM Activity
    WHERE activity_date BETWEEN DATE_SUB('2019-07-27', INTERVAL 29 DAY) AND '2019-07-27'
) AS distinct_users
GROUP BY day
ORDER BY day;

4️⃣ 复杂度分析

解法一：

  • ⏱️ 时间复杂度：O(n)，其中 n 是表中记录数。需要扫描一次表进行筛选和分组。
  • 💾 空间复杂度：O(m)，其中 m 是不同 (date, user_id) 组合数。分组操作需要临时存储。

解法二：

  • ⏱️ 时间复杂度：O(n)，同样需要扫描表，子查询可能增加额外开销。
  • 💾 空间复杂度：O(m)，子查询需要存储去重结果。

5️⃣ 解法对比

解法 优点 缺点 适用场景
解法一 简洁高效，单次扫描完成 逻辑稍紧凑 大多数情况，推荐使用
解法二 逻辑清晰，易于理解 可能有额外子查询开销 需要更易读的代码时

推荐：解法一。更简洁高效，符合 SQL 最佳实践。
